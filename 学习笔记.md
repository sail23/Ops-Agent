# 学习笔记：power-aiops-agents

> 面向电力数据中心运维的多智能体协作骨架 —— 全面入门指南

---

## 一、项目概述

一个面向**电力数据中心运维**场景的**多智能体协作框架**（Multi-Agent AIOps Skeleton）。通过 4 个虚拟专家 Agent（Ops / SRE / Code / Report）以编排流水线方式协同处理故障，覆盖从告警接入 → 初步诊断 → 深度分析 → 脚本执行 → 报告产出全流程。

核心定位是**骨架/脚手架** —— 核心流程已跑通，真实监控拉数、向量知识库等为桩/占位，便于后续扩展。

## 四、核心架构与数据流

### 4.1 请求流（Pipeline 模式）

```
用户/监控系统
    │
    ▼
POST /incidents/run  /  CLI power-aiops run --demo
    │
    ▼
build_incident_context()  ─── 构建 IncidentContext（事件、设备、摘要）
    │
    ▼
run_pipeline(ctx, board, memory)
    │
    ├── OpsAgent.run(ctx)
    │     → 初步诊断 + 任务分发
    │     → 写入 SharedBoard (ops_output)
    │
    ├── SREAgent.run(ctx)
    │     → 检索历史案例（Graph RAG + Vector RAG）
    │     → 深度分析方案
    │     → 写入 SharedBoard (sre_output)
    │
    ├── DynamicCodeAgent.run(ctx)
    │     → LLM 生成 Python 诊断代码
    │     → 安全围栏检查 (fence_check_text + _sanitize_code)
    │     → 可选执行代码 → 写入 SharedBoard (code_output)
    │
    └── ReportAgent.run(ctx)
          → 聚合所有 Agent 输出
          → 生成报告 + GPT-Vis 可视化数据
          → 写入 SharedBoard (report_output)
    │
    ▼
自动入库知识库（后台线程，LLM 提取症状/根因/解决方案）
    │
    ▼
IncidentRunResponse ← PipelineState
```

### 4.2 消息传递机制

| 组件 | 用途 | 实现 |
|---|---|---|
| **SharedBoard**（共享黑板） | Agent 间传递数据的核心通道 | 线程安全单例 + JSON 文件持久化（原子写入） |
| **ShortTermMemory**（短期记忆） | 记录 Agent 执行历史 | 固定窗口 deque（默认 20 轮） |
| **MessageLog**（消息日志） | 可选，记录 AgentMessage 信封 | `Protocol` 接口 + `InMemoryMessageLog` 实现 |
| **LongTermMemory**（长期记忆） | 持久化知识检索 | Graph RAG + Vector RAG 混合，RRF 融合 |

### 4.3 记忆系统（RAG）

```
LongTermMemory
    │
    ├── GraphRAG (Neo4j)
    │     ├── 节点：FaultCase, Symptom, Service, Host, RootCause, Resolution, Trace, Span
    │     ├── 关系：AFFECTS, HAS_SYMPTOM, CAUSED_BY, RESOLVED_BY, CONTAINS, CALLS
    │     └── 向量索引：symptom_embedding, rootcause_embedding
    │
    └── ChromaVectorRAG
          ├── Collection: incident_cases
          ├── 语义搜索（embedding → cosine distance → similarity）
          └── RRF 混合排序（Reciprocal Rank Fusion）
```

### 4.4 安全模型（三层防护）

| 层级 | 机制 | 位置 |
|---|---|---|
| **① 文本围栏** | 80+ 正则黑名单（rm -rf、drop table、eval、敏感路径、SQL 注入...） | `security/fences.py` |
| **② 代码静态分析** | 自定义 `_sanitize_code()` 黑名单（os.popen、eval、subprocess...） | `agents/dynamic_code.py` |
| **③ 角色权限矩阵** | 4 个 Agent × 6 个 Permission，运行时检查 | `security/permissions.py` |

## 九、扩展指南

### 如何添加一个新 Agent？

1. 继承 `BaseAgent`，实现 `agent_id` / `run()` / 可选 `stream_run()`
2. 在 `prompts/roles.py` 编写对应的 system prompt
3. 在 `orchestration/pipeline.py` 的 `_create_default_agents()` 注册
4. 在 `memory/shared_board.py` 添加 `BOARD_KEY_*` 常量（如有需要）

### 如何接入真实 Prometheus？

```bash
# .env 中配置
PROMETHEUS_MOCK_MODE=false
PROMETHEUS_BASE_URL=http://your-prometheus:9090
PROMETHEUS_BEARER_TOKEN=your_token_optional
```

`get_prometheus_client()` 自动根据配置返回 `PrometheusClient` 或 `MockPrometheusClient`。

### 如何切换 Embedding Provider？

修改 `llm/embedding.py` 中的 `ZhipuEmbeddingClient`，或在现有类中新增方法。外部调用通过 `embed_single()` / `embed_batch_with_fallback()` 接口解耦。

- **安全围栏系统**：开发双层安全检查——80+ 正则黑名单拦截高危命令 / SQL 注入 / 路径穿越，以及对 LLM 生成代码的静态分析器，所有代码执行受配置开关控制且默认关闭

## 十二、面试官视角的高频问题

> 以下是从面试官角度出发，针对本项目可能提出的考察点，涵盖架构设计、技术实现、安全、扩展性四个维度。

---

### 12.1 架构设计类

**Q1：为什么采用线性 Pipeline 而非并行 Agent？**

三个原因：① 故障诊断各步骤存在天然依赖关系——先诊断才能分析，先分析才能执行，先执行才能出报告；② 线性流保证了每个 Agent 输入的完整性（SRE 需要看到 Ops 的输出才能做根因分析）；③ 架构简单可控，便于后续加入人工审批节点。如果未来需要复杂辩论，可以切换到 `DebateOrchestrator`（已有实现）。

**Q2：SharedBoard 为什么设计为单例？如果部署多个 Worker 会有什么问题？**

单例是为了确保所有 Agent 在**同一个进程**中读写同一份内存数据。多 Worker（如 `uvicorn --workers N`）部署时每个 worker 有自己的进程内存空间，虽然文件持久化还能用，但会有**最终一致性问题**——Worker A 写入的数据 Worker B 不会立即看到。生产建议有两种方案：① 用 `--workers 1` + 多线程（SharedBoard 的 `RLock` 是线程安全的）；② 将 SharedBoard 替换为 Redis / 共享内存。

**Q3：为什么同时用图数据库和向量数据库做 RAG，只用一种不行吗？**

各有不可替代的优势：图 RAG 擅长**结构化关系查询**（如"某服务影响了哪些下游服务？上次类似拓扑的故障怎么解决的？"），向量 RAG 擅长**语义相似度匹配**（如"描述症状文本 → 检索相似案例"）。故障诊断中两种场景都会遇到，所以用 RRF（Reciprocal Rank Fusion）算法融合两类结果。

**Q4：Agent 之间如果产生了分歧，系统怎么处理？**

当前 Pipeline 模式下 Agent 是顺序执行的，后一个 Agent 可以引用、补充或质疑前一个 Agent 的输出（如 SRE 可以修正 Ops 的初步诊断）。系统还有**辩论模式**（`DebateOrchestrator`），支持多个 Agent 多轮辩论交锋，第二轮辩论时各 Agent 能看到对方上轮的输出并做出回应，最后 Report Agent 裁决。

### 12.2 技术实现类

**Q5：FastAPI 的 SSE 流式是怎么实现的？如果客户端断开连接会怎样？**

使用 FastAPI 的 `StreamingResponse` + `AsyncGenerator[AgentStreamChunk]`。每个 Agent 调用 `stream_run()` 时逐 token yield 数据，SSE 格式发送给前端。客户端断开时，`async for` 循环会抛出 `RuntimeError`，生成器退出，底层 httpx AsyncClient 连接自动关闭。当前没有做显式的断开检测回收（生产环境需要在 yield 时检查 `asyncio.Event` 信号）。

**Q6：pydantic v2 相比 v1 在这个项目中带来了什么实际好处？**

主要是两点：① 验证性能提升 5-50x（核心引擎用 Rust 重写），在 `IncidentContext` 批量解析 events 时能感受到差异；② `model_dump(mode="json")` 一行代码解决 datetime → ISO 字符串序列化问题，不需要手写转换 helper。

**Q7：LLM 调用的重试策略怎么设计的？**

指数退避，最多 3 次：`RETRY_DELAY = 3 × 2^attempt`。覆盖 `ReadTimeout`、`ConnectError`、`RemoteProtocolError` 三种网络异常，并对 HTTP 429（Rate Limit）做特殊处理（双倍退避）。但需要注意——同步的 `chat()` 调用目前使用 `time.sleep()`，在 FastAPI 异步路径中会阻塞事件循环，这是一个已知优化点。

**Q8：embedding fallback 到 hash 后搜索结果会变成什么样？**

SHA-256 哈希是确定性的但**不包含任何语义信息**。这意味着"CPU 使用率高"和"数据库连接超时"这两个完全不同的症状如果哈希值不同就不会匹配，只有"完全相同的文本"才会匹配同一个哈希。所以 semantic search 实际上是退化为 exact match，效果极差。系统启动时会打印一次 WARNING 日志提示使用者配置 Zhipu API key。

### 12.3 安全类

**Q9：LLM 生成的代码可能包含恶意指令，你们怎么防护的？**

三层防护：① `code_execution_enabled` 配置开关，**默认 false**；② `fence_check_text()` 正则围栏检查（80+ 高危模式，覆盖文件系统、数据库、网络、敏感路径、系统修改等）；③ `_sanitize_code()` 第二层静态分析器（检查 os.popen、eval、subprocess、socket 等危险函数调用）。即便如此，我们仍认为**沙箱隔离**是必须的（如 K8s Job / Docker 容器），这部分在项目规划中但尚未实现。

**Q10：正则黑名单会不会被绕过？**

会。这是正则静态分析的固有局限——攻击者可以通过 `split` + `concat`、base64 编码、间接函数调用等方式绕过。例如 `os.system("rm -rf /")` 可以写成 `getattr(os, "system")("rm -rf /")` 绕过当前规则。这也是为什么安全方针是"围栏 + 人工审核"而非"围栏替代人工"。

**Q11：Agent 超时怎么处理？**

每个 Agent 的 `run()` 是同步调用，目前**没有超时机制**。如果一个 Agent 的 LLM 调用卡住，整个 Pipeline 会阻塞。`DynamicCodeAgent` 的代码执行有 `execution_timeout` 超时（通过 `subprocess.run(timeout=...)` 控制），但 LLM `chat()` 本身依赖 http 超时（`openai_timeout_seconds` 配置）。合理的改进是对 Agent 套 `concurrent.futures.TimeoutError` 或 `asyncio.wait_for()`。

### 12.4 扩展性与演进类

**Q12：如果要增加第五个 Agent（比如安全审计 Agent），需要改多少代码？**

**Q14：怎么给这个项目写单元测试？Agent 依赖了 LLM 怎么 Mock？**

**Q15：监控数据量大的时候性能瓶颈可能在哪里？**

主要瓶颈在三个点：① LLM API 调用延迟（通常 1-5 秒，流式模式能改善首 token 体验但总时间不变）；② `SharedBoard.set()` 写文件（已改为批量写入单文件的原子操作，但高频调用仍有 I/O 开销）；③ 当前没有并发请求队列，多个 Pipeline 实例同时运行可能导致 LLM API 被限流。性能优化方向：LLM 调用改用连接池复用、SharedBoard 接 Redis、Pipeline 增加并发控制。
