## 阶段一：数据模型（models/）— 从外部事件到内存结构

### 核心文件

| 文件 | 核心概念 |
|------|---------|
| `models/events.py` | `EventSource` 枚举（prometheus/zabbix/elk/snmp/apm/manual），`EventObject` Pydantic 模型 |
| `models/incident.py` | `IncidentContext` Pydantic 模型（16 个字段），`update_from_pipeline()` 方法 |
| `models/messages.py` | `AgentMessage` 消息信封（sender/receiver/payload），`agent_message_from_json_dict` 工厂函数 |

### 为什么用 Pydantic？

项目选择 Pydantic 作为核心建模工具有三个原因：

1. **自动校验字段类型**：`EventSource` 枚举限制了告警来源只能是 6 种值之一，外部传入非法值时 Pydantic 自动抛 `ValidationError`。
2. **`.model_dump(mode="json")` 自动序列化**：SharedBoard 需要将 Agent 输出 JSON 持久化到文件，这一行搞定所有日期/dataclass 的转换。
3. **字段有默认值**：所有可选字段都有 `default_factory` 或默认值，代码中不需要大量 `if x is None` 的判断。

### `EventObject` — 统一告警格式

```python
# src/power_aiops/models/events.py
class EventObject(BaseModel):
    timestamp: datetime
    device_id: str = Field(..., description="Host / asset id")
    metric_type: str = Field(..., description="e.g. cpu_usage, log_error_rate")
    value: str | float | int | None = None
    raw_payload: dict[str, Any] = Field(default_factory=dict)
    source: EventSource = EventSource.MANUAL
```

核心设计：**`raw_payload: dict` 保留原始数据**。Prometheus 返回的 JSON 格式、Zabbix 的 XML 格式全部塞进这个字段，转换层不需要知道具体字段，只需要把它规范化为这 5 个统一字段。

### `IncidentContext` — 全流程上下文

```python
# src/power_aiops/models/incident.py
class IncidentContext(BaseModel):
    incident_id: str
    trace_id: str
    created_at: datetime = Field(default_factory=lambda: datetime.now(timezone.utc))
    events: list[EventObject] = Field(default_factory=list)
    summary: str = ""

    # Pipeline-filled fields
    title: str = ""
    description: str = ""
    symptoms: list[str] = Field(default_factory=list)
    root_cause: str = ""
    resolution: str = ""
    severity: str = ""

    # Service info (optional)
    service_name: str = ""
    occurred_at: datetime | None = None

    # Metadata
    shared_notes: list[str] = Field(default_factory=list)
    metadata: dict[str, Any] = Field(default_factory=dict)

    def update_from_pipeline(...): ...
```

注意这个类的两个阶段：

- **进入 Pipeline 前**：只有 `incident_id / trace_id / events / summary` 被填充
- **Pipeline 执行中**：每个 Agent 依次填充自己的字段（`symptoms`、`root_cause`、`resolution` 等）

`update_from_pipeline()` 方法体现了**渐进式填充**思想：不需要一次性知道所有信息，每个 Agent 完成自己的分析后更新进去即可。

### `AgentMessage` — Agent 间消息信封

```python
# src/power_aiops/models/messages.py
class AgentMessage(BaseModel):
    msg_id: str = Field(default_factory=lambda: str(uuid4()))
    timestamp: datetime = Field(default_factory=lambda: datetime.now(timezone.utc))
    sender: AgentRef
    receiver: AgentRef
    priority: MessagePriority = MessagePriority.NORMAL
    context_ref: str = Field(..., description="Incident id")
    trace_id: str = Field(..., description="End-to-end fault handling trace")
    payload: MessagePayload = Field(default_factory=MessagePayload)
    deadline_ms: int | None = Field(default=None, description="Optional SLA for response")
```

`trace_id` 贯穿整个故障处理过程，所有 Agent 的消息都有这个 ID，便于事后回放整个处理链路。`MessagePayload` 里有个 `extra: dict` 字段存放 Agent 专属数据（如代码执行结果），不污染消息结构。

### 练习思考

> **如果要从 Zabbix 接入告警，需要在哪一步做转换？**

答：在 API 入口 `build_incident_context()`（`api/schemas.py`）或独立的 `integrations/zabbix.py` 中将 Zabbix 的 webhook payload 映射为 `EventObject[]`，其余所有 Agent/Pipeline 代码无需任何改动——这就是统一建模的好处。

## 阶段二：四大 Agent（agents/）— 各自的职责与协作约定

### Agent 基类设计

```python
# src/power_aiops/agents/base.py
class BaseAgent(ABC):
    @property
    @abstractmethod
    def agent_id(self) -> str: ...

    @abstractmethod
    def run(self, ctx: IncidentContext) -> AgentResult: ...

    async def stream_run(self, ctx: IncidentContext) -> AsyncGenerator[AgentStreamChunk, None]:
        result = self.run(ctx)
        yield AgentStreamChunk(
            agent_id=self.agent_id, delta=result.content,
            is_done=True, blocked=result.blocked,
            fence_matched=result.fence_matched,
        )
```

`BaseAgent` 定义了**同步 run 和异步 stream_run 两个接口**。默认 `stream_run` 只是包装 `run()` 一次输出一个大块（`is_done=True`），但每个子类都覆盖了真正的 token 级流式实现。

### OpsAgent — 运维专家

```python
# src/power_aiops/agents/ops.py
def run(self, ctx: IncidentContext) -> AgentResult:
    user = self._user_prompt(ctx)
    if self._use_llm:
        text = self._llm.chat(system=SYSTEM_PROMPT_OPS_AGENT, user=user)
        meta = {"llm": "openai-compatible" if self._llm.is_configured() else "stub"}
    else:
        text = self._placeholder(ctx)
        meta = {"llm": "stub"}

    self._board.set(BOARD_KEY_OPS, text)  # 写入 SharedBoard
    return AgentResult(agent_id=self.agent_id, content=text, meta=meta)
```

关键设计：**`use_llm=False` 时走固定逻辑模板，`use_llm=True` 时才调用 LLM**。默认 `use_llm=True`（`__init__` 参数）。配置 DeepSeek 或 OpenAI API Key 后，所有 Agent 自动切换到真实 LLM 调用，无需改代码。

### SREAgent — 混合 RAG 的架构师

这是整个项目设计最亮眼的部分。SREAgent 在做架构分析**之前**，会先检索相似的历史故障案例：

```python
# src/power_aiops/agents/sre.py

            incident=ctx, top_k=self._top_k,
            graph_weight=0.4, vector_weight=0.6,

        # 格式化检索结果注入 prompt...
```

检索结果中同时展示了 **Graph 和 Vector 两个来源的分数**，让分析者知道这个案例是"图关系匹配到的"还是"语义相似匹配到的"。

## 阶段三：记忆系统（memory/）— 从短期到长期的知识管理

### SharedBoard — 单例 + 文件持久化

```python
# src/power_aiops/memory/shared_board.py
class SharedBoard:
    _instance = None
    _instance_data: dict[str, Any] = {}
    _instance_lock = RLock()  # 线程安全
    _persist_dir = "data/shared_board"

    def __new__(cls):
        if cls._instance is None:
            with cls._instance_lock:
                if cls._instance is None:
                    cls._instance = super().__new__(cls)
        return cls._instance

    def set(self, key: str, value: Any) -> None:
        with self._lock:
            self._instance_data[key] = value
            persist_path = self._get_persist_path(key)
            with open(persist_path, 'w', encoding='utf-8') as f:
                json.dump(value, f, ensure_ascii=False, indent=2)
```

每个 `set()` 调用同时写入内存字典和 JSON 文件。`get()` 时会尝试从文件恢复，支持**进程重启后数据不丢失**。

**关键设计**：为什么是单例？因为 FastAPI 多 Worker 进程共享不了内存。注释里也提到：需要 Redis 才可真正跨进程共享。

### ShortTermMemory — 滑动窗口

```python
# src/power_aiops/memory/short_term.py
class ShortTermMemory:
    def __init__(self, max_turns: int = 20) -> None:
        self._max = max_turns
        self._turns: deque[Turn] = deque(maxlen=max_turns)

    def append(self, agent_id: str, content: str) -> None:
        self._turns.append(Turn(agent_id=agent_id, content=content))

    def summary_placeholder(self) -> str:
        """Wire to LLM summarization later."""
        return f"[summary of {len(self._turns)} turns — plug LLM here]"
```

滑动窗口默认 20 轮，超过后自动丢弃最老的记录。`summary_placeholder()` 是预留的 LLM 压缩接口。

### LongTermMemory — RRF 融合算法

```python
# src/power_aiops/memory/long_term.py
def _reciprocal_rank_fusion(self, graph_results, vector_results,
                            graph_weight, vector_weight, top_k):
    k = 60  # RRF constant
    graph_ranks: dict[str, float] = {}
    for rank, result in enumerate(graph_results, start=1):
        incident_id = result.get("incident_id", "")
        graph_ranks[incident_id] = graph_weight / (k + rank)

    vector_ranks: dict[str, float] = {}
    for rank, result in enumerate(vector_results, start=1):
        incident_id = result.get("incident_id", "")
        vector_ranks[incident_id] = vector_weight / (k + rank)

    all_ids = set(graph_ranks.keys()) | set(vector_ranks.keys())
    for incident_id in all_ids:
        graph_score = graph_ranks.get(incident_id, 0.0)
        vector_score = vector_ranks.get(incident_id, 0.0)
        combined_scores[incident_id] = {
            "graph_score": graph_score,
            "vector_score": vector_score,
            "total_score": graph_score + vector_score,
        }
```

RRF 公式：`score = weight × 1 / (k + rank)`，其中 `k=60`。两个检索源各自给出一个排名列表，综合得分最高的排在前面。

### GraphRAG — Neo4j 图谱

图谱结构设计为两类节点：

```
(FaultCase)-[:AFFECTS]->(Service)
           -[:HAS_SYMPTOM]->(Symptom {embedding})   ← 向量索引
           -[:CAUSED_BY]->(RootCause {embedding})   ← 向量索引
           -[:RESOLVED_BY]->(Resolution)
           -[:RELATED_TO*1..2]->(FaultCase)         ← 案例关联

(Trace)-[:CONTAINS]->(Span)-[:CALLS]->(Span)       ← 链路追踪
      -[:RELATED_TO]->(FaultCase)
```

链路追踪是独立于故障案例的另一套图谱，通过 `trace_id` 关联到具体案例。`get_fault_propagation_path()` 返回错误 Span 按时间排序的列表，展示故障传播路径。

### VectorRAG — Chroma 向量库

```python
# src/power_aiops/memory/vector_rag.py
def _combine_text(self, incident: IncidentDocument) -> str:
    parts = [
        f"标题: {incident.title}",
        f"服务: {incident.service_name or '未知'}",
        f"严重程度: {incident.severity or '未知'}",
        f"症状: {'; '.join(incident.symptoms)}",
        f"根因: {incident.root_cause}",
        f"解决方案: {incident.solution}",
    ]
    return " | ".join(parts)
```

向量搜索前把所有字段拼接为一个长字符串做 embedding，保证了语义完整性。fallback 用 SHA256 hash 生成伪向量，Zhipu Embedding API 不可用时仍能工作。

### 练习思考

> **如果历史故障案例中没有相似案例，`hybrid_search` 返回什么？`graph_rag.py` 的 `search_similar_cases()` 对 `str` 类型如何降级处理？**

答：Graph 返回空列表 `[]`，Vector 也返回空列表，RRF 融合后 `combined_scores` 为空字典，最终 `sorted_results = []`。SREAgent 的 `_retrieve_similar_cases()` 检测到 `if not results` 后直接返回空字符串，不影响主流程——**优雅降级**。

`search_similar_cases()` 收到 `str` 类型的 case 时，直接将其作为搜索查询文本，调用 `vector_search(case, search_type="symptom", top_k=top_k)` 做语义检索。

## 阶段四：编排引擎（orchestration/）— Pipeline 顺序执行 vs Debate 动态辩论

### Pipeline — 顺序流水线

```python
# src/power_aiops/orchestration/pipeline.py
def run_pipeline(ctx, *, board=None, memory=None,
                long_term_memory=None, message_log=None, agents=None):
    agents = (
        OpsAgent(board=board),
        SREAgent(board=board, long_term_memory=long_term_memory),  # ← 注入 RAG
        _create_code_agent(board),                                  # ← DynamicCodeAgent
        ReportAgent(board=board),
    )

    for label, agent in zip(("ops", "sre", "code", "report"), agent_list):
        result = agent.run(ctx)
        state.agent_outputs[result.agent_id] = result.content
        state.completed_steps.append(label)
        memory.append(result.agent_id, result.content)

        if label == "code":
            state.code_blocked = result.blocked
            board.set(BOARD_KEY_CODE_BLOCKED, result.blocked)

        if message_log:
            message_log.append(AgentMessage(...))  # 消息持久化
```

**设计亮点**：Code 被围栏拦截时 `state.code_blocked=True`，但 Pipeline 仍继续执行 Report。故障不能因为脚本有问题就不出报告。

**流程图**：

```
IncidentContext
    ↓
OpsAgent  → board["ops_output"]
    ↓
SREAgent → board["sre_output"]  （可选：混合 RAG 检索历史案例）
    ↓
CodeAgent → board["code_output"] （可选：代码执行 + fence 拦截）
    ↓
ReportAgent → board["report_output"]  （报告生成）
    ↓
PipelineState（所有 Agent 输出汇总）
```

### Debate — 动态辩论编排

这是比 Pipeline 复杂得多的设计。核心变化：**Agent 发言后输出 `next_turn` 指令，编排器解析后决定下一个发言者。**

```python
# src/power_aiops/orchestration/debate.py
def parse_next_turn(text: str) -> NextTurnHint:
    m = _NEXT_TURN_RE.search(text)  # 正则匹配 ## next_turn 区块
    if not m:
        if "converge" in raw or "收敛" in raw:
            return NextTurnHint.CONVERGE
        ...
    hint_text = m.group(1).strip().lower()
    if "sre" in hint_text and "converge" not in hint_text:
        return NextTurnHint.SRE
    ...
```

**收敛判断是双层**：

1. **算法层**：Jaccard 分数 ≥ 0.7 自动进入 Report
2. **LLM 层**：Agent 输出 `## convergence: true` 也触发收敛

```python
# src/power_aiops/orchestration/debate.py
def _update_convergence(self) -> None:
    tokens_a = set(self.ops_position.split())
    tokens_b = set(self.sre_position.split())
    tokens_c = set(self.code_position.split())
    def jaccard(a, b):
        return len(a & b) / max(len(a | b), 1)
    overlap_ab = jaccard(tokens_a, tokens_b)
    overlap_bc = jaccard(tokens_b, tokens_c)
    overlap_ac = jaccard(tokens_a, tokens_c)
    self.convergence_score = round((overlap_ab + overlap_bc + overlap_ac) / 3, 3)
```

### DebateBoard vs SharedBoard

| | SharedBoard | DebateBoard |
|--|-----------|-------------|
| 语义 | 当前/latest 状态 | 完整辩论历史 |
| 操作 | 覆盖写入 | 只追加 |
| 用途 | 各 Agent 共享最新输出 | 辩论全流程记录 |

```python
# src/power_aiops/orchestration/debate_board.py
def append(self, message: DebateMessage) -> None:
    with self._lock:
        self._messages.append(message)  # ← 只能追加，不覆盖
        self._latest_by_role[message.role] = message
```

### 练习思考

> **Debate 模式的报告从哪里来？是从 REPORT 角色的最终发言中提取的吗？**

答：从 `DebateOrchestrator._build_result()` 中提取。优先取 `round == "report"` 的 turn 的 `message.content`（即 Report-Agent 的最终裁决文本）。如果没有则取最后一个 turn 的内容。报告中的 `final_report` 字段由 `DEBATE_PROMPT_REPORT_JUDGE` 模板约束 LLM 输出。

### 自动入库流程

Pipeline 执行完成后，调用 LLM 从各 Agent 输出中提取结构化信息：

```python
# src/power_aiops/run_incident.py
def _extract_with_llm(incident_id, summary, agent_outputs) -> dict:
    prompt = _EXTRACTION_PROMPT_TEMPLATE.format(
```

提取的 JSON 包含 `symptoms / root_cause / resolution / services / hosts / tags`，同时存入 Graph RAG（Neo4j）和 Chroma Vector RAG。以后类似故障发生时，SREAgent 检索时可命中这些案例。

**自动入库流程图**：

```
Pipeline 执行完成
    ↓
```

### 导出复用设计

Pipeline 导出 (`/run/export`) 时，将 `PipelineState` 转换为 `DebateResult` 格式，重用了 `export_debate_report()`：

```python
# src/power_aiops/api/routes.py
# 将 Pipeline 结果转换为 DebateResult 格式以复用导出器
agent_outputs = pipeline_data.get("agent_outputs", {})
debate_result_dict = {
    "incident_id": incident_id,
    "total_turns": len(agent_outputs),
    "report_text": pipeline_data.get("report_content", ""),
    "history": {"turns": turns},
    ...
}
result = DebateResult(**debate_result_dict)
file_bytes = export_debate_report(result, format=format_type)
```

这个设计避免了为 Pipeline 导出写一套独立的 Word/PDF 生成逻辑。

### 练习思考

> **`/run/export` 接口的 Pipeline 结果是如何被转换为 `DebateResult` 格式再导出的？重用了哪一套导出器？**

答：`post_pipeline_export()` 将 `PipelineState` 中的 `agent_outputs` 按 agent_id 拆解为 `turns` 列表（每个 Agent 占一个 turn），构建 `DebateResult` 字典后用 `**` 解包构造 `DebateResult` 实例。重用了 `export/debate_report_exporter.py` 中的 `export_debate_report()` 和 `ReportExporter` 类，支持 docx 和 pdf 两种格式。

## 关键设计总结

| 设计点 | 实现位置 | 核心思想 |
|--------|---------|---------|
| Pydantic 统一建模 | `models/` | 外部多样化 → 内部统一 |
| SharedBoard 单例 + 文件持久化 | `memory/shared_board.py` | 进程内共享 + 跨进程恢复 |
| 渐进式填充 IncidentContext | `models/incident.py` | 每步 Agent 只填自己的字段 |
| RRF 混合检索融合 | `memory/long_term.py` | 排名融合而非分数融合 |
| LLM 生成代码 + Neo4j 查询 | `agents/dynamic_code.py` | 避免向 LLM 发大量上下文 |
| 围栏 + subprocess 隔离 | `security/fences.py` + `dynamic_code.py` | 纵深防御 |
| `next_turn` 指令驱动辩论路由 | `orchestration/debate.py` | LLM 自举决定辩论走向 |
| 自动入库 + LLM 提取结构化信息 | `run_incident.py` | 闭环学习 |
